    1. Create `app/api/ingest/route.ts` -- POST endpoint:
       a. Parse JSON body. If parse fails, return 400 with error.
       b. Validate body with ingestionSchema.safeParse(). If invalid, return 400 with field-level errors: { error: 'Validation failed', details: issues.map(i => ({ field: i.path.join('.'), message: i.message })) }.
       c. Extract x-api-key header (may be absent on first push). Extract accountId from validated body.
       d. Authentication + auto-registration flow:
          - Query DB for an existing account with the given accountId.
          - **Case 1 — New account (no record for accountId):** This is a first push. Auto-register: create new account record with accountId as id, accountName as display_name, generate new API key via generateApiKey(), set primary_kpi to 'cpa'. Continue to ingestion. Return the generated API key in the response so the user can update their script.
          - **Case 2 — Existing account, valid key (x-api-key matches stored api_key):** Authenticated. Continue to ingestion.
          - **Case 3 — Existing account, wrong/missing key (x-api-key absent or doesn't match stored api_key):** Return 403 { error: 'Invalid API key for this account', details: [] }.
       e. For each campaign type array in the payload (rsa, rsaAssets, pmax, pmaxAssets, display, video):
          - Map records to include account_id and date from top-level payload
          - Use Drizzle insert().values().onConflictDoUpdate() with the composite unique target
          - The onConflictDoUpdate set clause should update ALL metric columns with `sql\`excluded.column_name\`` syntax (see research upsert pattern)
       f. Update account last_synced_at to now()
       g. Create sync_log entry: { account_id, pushed_at: now(), record_count: total records across all types, status: 'success' }
       h. If any database operation fails, catch the error, create sync_log with status: 'error' and error_message, return 500
       i. Return 200 { status: 'ok', recordCount, accountId, apiKey?: (only on first registration) }

    2. Create `app/api/accounts/route.ts` -- GET endpoint:
       - Query all accounts with their last 10 sync_log entries
       - Return { accounts: [{ id, displayName, primaryKpi, lastSyncedAt, recentSyncs: [{ pushedAt, recordCount, status, errorMessage }] }] }
       - No authentication required for v1 (single operator, per user decision)

    3. Create `app/api/accounts/[id]/route.ts` -- GET endpoint:
       - Query single account by id with full sync_log (last 10 entries)
       - Return 404 if account not found
       - Return account details plus recentSyncs array

    IMPORTANT: Use `import { db } from '@/lib/db'` for database access. Use `import * as schema from '@/lib/db/schema'` for table references. Use `import { eq, and, desc } from 'drizzle-orm'` for query helpers. Use `import { sql } from 'drizzle-orm'` for raw SQL in onConflictDoUpdate set clauses.

    AUTO-REGISTRATION FLOW: On first ever push, the script won't have an API key yet. The flow is:
    1. Script sends POST with accountId and accountName but no x-api-key header (or an empty/placeholder value)
    2. API parses + validates the body first (returns 400 if malformed)
    3. API checks if accountId exists in DB. It doesn't → Case 1 (new account): creates account record, generates API key
    4. API ingests the data, creates sync_log entry, returns: { status: 'ok', apiKey: 'ca_xxxxx', message: 'Account registered. Update your script with this API key.' }
    5. User updates script CONFIG.API_KEY with the real API key
    6. Subsequent pushes include x-api-key header → Case 2 (valid key): authenticated normally
    7. If someone sends a request with a wrong key for an existing account → Case 3: 403 rejected
