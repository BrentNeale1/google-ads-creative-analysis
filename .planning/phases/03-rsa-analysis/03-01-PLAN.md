**Schema changes (lib/db/schema.ts):**

1. Add `textContent` column to `rsaAssetDaily` table:
   ```typescript
   textContent: text('text_content'),  // nullable -- existing data won't have it
   ```
   Place it after `fieldType` and before `performanceLabel`. This stores the actual headline/description text from `asset.text_asset.text`.

2. Add new `rsaCombinationDaily` table:
   ```typescript
   export const rsaCombinationDaily = pgTable(
     'rsa_combination_daily',
     {
       id: serial('id').primaryKey(),
       accountId: text('account_id').references(() => accounts.id).notNull(),
       date: date('date').notNull(),
       adId: text('ad_id').notNull(),
       /** Headline texts in this combination, JSON array of strings */
       headlines: jsonb('headlines').notNull(),
       /** Description texts in this combination, JSON array of strings */
       descriptions: jsonb('descriptions').notNull(),
       /** Impressions only -- Google does not provide clicks/conversions for combinations */
       impressions: integer('impressions').default(0).notNull(),
     },
     (table) => [
       uniqueIndex('rsa_combo_daily_account_date_ad_combo_idx').on(
         table.accountId,
         table.date,
         table.adId,
         // Use a hash or just let the combo of ad+headlines+descriptions be unique enough
         // Actually need more specificity -- use sql expression or just accountId+date+adId+id
       ),
     ],
   );
   ```
   Note: Since combinations are identified by their headline+description content (JSONB), use a simpler unique index on `(accountId, date, adId)` -- Google reports one "top" combination per ad per day. If multiple combos per ad per day are possible, remove the unique index and rely on upsert logic with a content check. For now, store top combinations per ad per day.

   Actually, per the research, combination data from `ad_group_ad_asset_combination_view` can have multiple combinations per ad. Use a composite unique index on `(accountId, date, adId, id)` -- or better, just don't use a unique index and do simple inserts (delete+insert pattern for combinations on each sync). Simplest approach: unique index on `(accountId, date, adId)` is wrong if multiple combos exist per ad. Use NO unique index and delete-before-insert in the ingestion route for combinations.

   Final approach: No unique index on rsaCombinationDaily. The ingestion route will delete existing combination rows for that account+date before inserting new ones. This is clean because combination data is directional and fully replaced on each sync.
